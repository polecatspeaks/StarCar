## 2. The manifest contract

- **YB-1.** A train manifest is an `intent` record whose `subject` matches
  `^train:[a-z0-9][a-z0-9-]*$`, carrying the single new top-level key `manifest`
  (`title`, `tickets[]`, `members[]{subject, role, gate?}`), per
  `schema/starcar-manifest.schema.json`. Store validators (CI's `StoreIntegrity` layer)
  run this schema IN ADDITION to the base record schema; neither restates the other.
- **YB-2.** `schema/vocab/roles.json` (landed at this rung) is the role recognition
  vocabulary: `car`, `reviewer`, `gate` - exactly the design's set. `[SB-7, folded]`:
  an earlier draft shipped `conductor`, unobserved and uncontracted; dropped, because
  YB-3's epistemic rule binds roles too - the detector discovers it if it ever appears. **Deviation from design, disclosed:** the design illustrated roles as
  single tokens (`gate:<name>`); this spec splits the gate NAME into `members[].gate`
  so the recognition vocabulary stays unparameterised and the detector's
  known/discovery distinction works unchanged. Folds back into the design as an
  amendment block on approval of this spec.
- **YB-3.** `schema/vocab/outcomes.json` gains `done` and `CONFIRM` - both OBSERVED in
  the live store (5 and 3 records respectively, enumerated 2026-07-23; the detector
  fires discoveries on them today) - and `done-with-findings`, enumerated by the
  envelope CONTRACT (`docs/templates/car-brief.md`'s outcome set). Nothing else is
  added: unobserved, uncontracted values wait for the detector to discover them
  (design §2's epistemic constraint). The detector test asserting the vocabulary
  updates in the same commit.

## 3. The wire contract

- **YB-4.** `schema/yard-snapshot.schema.json` (landed at this rung) owns the wire
  SHAPE. The Go server's marshal path and the browser's validator both conform to it;
  any hand-maintained mirror is a finding. The SSE event name is
  `$defs.sseEventName.const` - server writer and client subscriber each carry a test
  asserting against the schema's constant, not against a local string.
- **YB-5.** Lane payload shapes are `$defs` EXTENSIONS to the wire schema, landed
  red-first by the schema car. Required content (the requirement; the schema is the
  contract once landed):
  - `trainsPayload`: `trains[]{ id (whole train: subject), title, cars[]{ subject,
    role, gate?, state, at, outcome?, superseded? } }` - each car entry joins the
    WINNING manifest's member declaration with that subject's fold output; plus
    `declaredNotObserved[]` for members with no records (design §6).
  - `gatesPayload`: `gates[]{ name (from member gate/role declaration), subject,
    outcome (VERBATIM), at }`.
  - `dispatchesPayload`: `dispatches[]` mirroring the fold's dispatch entry shape
    (the vectors own that shape), plus `assigned: boolean` per entry (yard inventory =
    unassigned, rendered loudly).
- **YB-6.** Composition semantics (design §5.2 Rules 1-4: most-severe-wins across
  THREE axes, the freshness→register mapping, position-primary/freshness-secondary,
  server-issued quantised age, detector register) are pinned by an EXHAUSTIVE
  **three-axis** matrix test in the view car `[SB-8, folded]`: every position register
  x every freshness kind x both capability states (renderer present / absent - absent
  contributes `needs-attention`, design Rule 1), with two load-bearing cases red
  against partial implementations: `live` + `failed` → `needs-attention`, and
  `live` + `fresh` + no-renderer → `needs-attention` with the "no renderer for this
  payload" line.
- **Naming deviation, disclosed:** the design carried `statePathDisplay` (a rev 3
  relic naming the retired state file); the wire schema names it `storePathDisplay`
  (it displays the STORE path). Folds back as a design amendment with YB-2's.

## 4. The fold port and its vectors

- **YB-7.** The Go fold conforms to every vector in `schema/vectors/fold/` under the
  runner contract in `schema/vectors/README.md` (materialise records, inject vocab and
  clock, deep-equal on `faults`/`discoveries`/`dispatches`/`intents`).
- **YB-8.** `empty-vocab-one-fault.json` is DESIGN-MANDATED and red-on-arrival: the
  landed pwsh detector provably fans out on its input (observed: 4 false discoveries,
  0 faults). The scoped fix edits `scripts/Detect-Dispatches.ps1` to emit **ONE
  combined fault, identical in shape to the malformed-vocabulary fault, naming every
  empty file** (`vocabulary: valid but empty: <files, alphabetical>`) and zero
  per-record discoveries `[DR4-2, folded; SB-4, folded: an earlier draft's per-file
  faults deviated undisclosed from the design's one-fault-identical-to-malformed rule]`
  - red-first against this vector, in its own commit, with `Detector.Tests.ps1`
  gaining the matching case. **Disclosed posture `[SB-6, folded]`:** fault strings
  pinned by vectors are cross-language contract surfaces - both implementations emit
  them byte-identically. Scoped to vector-covered cases; the detector's other freeform
  fault strings acquire this obligation only as vectors cover them (structured fault
  codes are the escape hatch if string-pinning proves brittle, recorded as a
  trigger-gated alternative, not built now).
- **YB-9.** The D18 cross-verifier is a REAL CI job: one step runs BOTH the pwsh
  detector and the Go fold against every `schema/vectors/fold/` vector and fails on
  any divergence. It is not done when green: it is done when an INJECTED divergence
  has been WATCHED to fail it (fault-inject, observe red, revert, record the run URL) -
  the Q1 binding condition.
- **YB-10.** The existing `Detector.Tests.ps1` imperative cases migrate INTO
  `schema/vectors/fold/` as declarative fixtures (Q8 ruling), red-first, proven by the
  existing assertions passing green against the SAME cases rehomed - its own reviewed
  task, never bundled into the Go-port car.

## 5. Server, view, validators

- **YB-11.** Two plan-rung BLOCKING TESTS before any car depends on them (design §2c):
  (a) a Go draft-2020-12 validator library validates the landed store schema **AND the
  manifest schema** `[SB-1, folded]` + their vectors, observed; (b) a no-build-step
  browser JS draft-2020-12 validator validates the wire schema, observed in a bare
  browser context. Each has its disclosed degradation branch (structural checks, named
  in the gating matrix, with the validator's arrival as retirement trigger).
  **Standing rule `[SB-1, folded]`:** every `pattern` in every landed schema is
  RE2-compatible - no lookahead, no backreferences; the Go-compile probe
  (`scripts/probes/go/` pattern check) is the mechanical check, and a
  non-RE2-compatible pattern is a defect wherever it appears.
- **YB-12.** Server behaviours carried from the design, each landed red-first: poll
  loop with skip-not-queue overlap guard; change detection excluding `seq`/`asOf` and
  including `freshness.kind`/`ageBucketMs`, `seq` assigned after comparison; SSE
  heartbeat with client flip after two missed; monotonic `seq` application
  client-side; `storePathDisplay` never a raw absolute path.
- **YB-13.** Store-read path per the design: per-record double-decode
  (typed + `RawMessage` key-diff), quarantine-with-disclosure, unknown fields
  preserved and disclosed, the §6 failure table's rows each landed as a test against
  its fixture. The design's TWO distinct collisions each get their own named fixture
  `[SB-9, folded]`: (a) the **DR3-3 namespace collision** - one subject in both
  `fold.dispatches` and `fold.intents` - raises board condition
  `subject-namespace-collision`; (b) the **Q2 membership collision** is YB-14's.
  Other rows: empty store, all-quarantined, mid-write partial, future-dated `at`,
  manifest-names-unobserved-subject.
- **YB-14.** `[SB-2, folded - the dropped Q2 adopted ruling, restored]` When two
  different manifests both claim one dispatch subject, the Assembler renders the
  dispatch in BOTH trains AND raises board condition `manifest-membership-collision`
  naming the dispatch and both train subjects - disclosed, never resolved by silent
  precedence (latest-`at` is same-subject store law and does NOT extend across
  different manifest subjects; design §6 `[Q2, ruling adopted]`, Laws 4 and 6).
  Red-first against a two-manifests fixture in the Assembler's suite.
- **YB-15.** `[SB-3, folded - required-but-undefined wire field, now defined]`
  `config.demoMode` is true exactly when the server was started against a demo
  fixture store (a curated fixture directory instead of a live checkout's
  `artifacts/`). Its SOLE consumer is the view's persistent DEMO banner - an
  additional honest disclosure. **No other behavior may branch on it**: a truth
  surface (staleness, disconnect, detector, board conditions) suppressed or altered
  under `demoMode` is a gating-matrix violation (the design retired demo-mode
  suppression; this field exists to DISCLOSE demo data, never to mute anything).
  Red-first: a test asserting the banner renders when true, plus a test asserting
  staleness still fires on stale demo data.
