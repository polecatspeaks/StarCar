## MAJOR FINDINGS (any one = REJECT)

### MAJOR-1. The `LaneState` union has no inhabitant for "not yet known", and Law 1 requires one

**Location:** design:73-77 (the union), design:100-108 (`Server`), design:172-179 (error table).

The union is `ok | stale | failed | no-adapter`. Now consider the window the design never names: the server has started, the poll loop has not completed its first pass, and a browser hits `GET /api/snapshot` (design:103). What is returned?

- `ok` requires `data` and is a positive claim of freshness. Unbacked.
- `stale` requires `data`. Unbacked.
- `no-adapter` is a **lie**: there is an adapter, it just has not run.
- `failed` requires a `reason`, and there has been no failure.

There is no fifth option, and `YardSnapshot.asOf` (design:80) also has no defined value before the first poll. `constitution.md:17` is explicit: **"Unknown states render AS unknown, honestly."** The design's proudest structural move (design:89-93, "Law 4 becomes a type error rather than a review rule") builds a closed union that cannot express the one state the first law names by hand.

**Why it matters:** four cars will each invent a different answer. Car C returns *something* from `/api/snapshot`; Car A's assembler must construct it; Car D must render it. Whatever Car C picks, the board's first paint is a confident falsehood on a status surface. That is the exact defect `constitution.md:15-16` names as the worst this project can ship.

**Fix:** add `| { kind: 'unknown' }` (or `'pending'`) to `LaneState`, make `YardSnapshot.asOf` nullable or add a `firstPollCompleted` discriminant at snapshot level, define the initial snapshot explicitly in §3.4, and add "the pre-first-poll window" to the §7 lifecycle enumeration. Also add a §6 test: "before the first poll completes, `/api/snapshot` returns every lane as `unknown` and the view renders 'not yet read', never an empty lane and never `no-adapter`."

### MAJOR-2. The sentence-check reduction is unsound, and §6's shared-type test cannot fail

**Location:** design:30 (D2), design:164-168, design:195-196.

design:167-168 claims the sentence check "reduces to confirming the shared type is genuinely shared (one definition, imported both sides) rather than tracing nine fields through sixteen files." design:195-196 claims the guard is mechanical: "A duplicate definition is a build failure, not a review finding."

Three independent legs, each fatal.

**(a) A duplicate definition is not a build failure. TypeScript is structurally typed.** A second `type YardSnapshot = {...}` in the view module compiles cleanly, interoperates cleanly with the server's, and produces no diagnostic of any kind. Even a *divergent* duplicate compiles, as long as no value is assigned across the two definitions, which is exactly the situation here because the value arrives from `JSON.parse` and is cast. `tsc` has no "this type is defined twice in the repo" check, no lint rule enables one by default, and no bundler reports it. **The mechanical guard that D2's entire rationale rests on does not exist.** The test as written at design:195-196 would pass the moment it is written, against a codebase with two divergent definitions, which `CLAUDE.md:99` names outright: "A test that passes the moment you write it proves nothing."

**(b) `JSON.parse` on the client is an unchecked cast, and the schema module's validators are not applied at the boundary.** design:147 says the schema module defines "`Yard`, `Train`, `Car`, `Gate`, and their validators." design:63-64 applies validation on adapter read. design:139-141 applies validation on write. **No validation is applied to the wire payload.** The client does `JSON.parse(e.data) as YardSnapshot` and TypeScript believes it. A shared compile-time type constrains what the server's *source code claims* to send and what the client's *source code claims* to receive. It constrains nothing about what crosses the wire. Concretely, in this design:
- `ageMs: number` (design:75). If a clock skew or a division produces `Infinity` or `NaN`, `JSON.stringify` emits `null`. TypeScript still says `number`. The view renders `null ms` or `NaN`.
- `lastGood?: T` (design:76). `JSON.stringify` silently drops keys whose value is `undefined`, so an explicitly-cleared `lastGood` and an absent one are indistinguishable on the wire. If any car ever reaches for `Date`, `Map`, `Set`, `bigint`, or a class instance inside `Train`/`Gate`/`Fuel`, the type says one thing and the wire says another, with no diagnostic.

This is precisely the class `CLAUDE.md:150-161` was paid twice for. The design does not delete the class; it renames it and declares the renamed thing safe.

**(c) The SSE layer is a hand-written string protocol that no type covers.** design:104 says `/api/stream` "emits a snapshot event when the snapshot changes." An SSE *event name* is a hand-maintained string on both sides (server writes `event: snapshot\ndata: ...\n\n`; client writes `addEventListener('snapshot', ...)`). If they disagree by one character, the client's listener never fires, the connection stays **open**, and per design:178 the client therefore reports itself **connected** while displaying the first-paint snapshot forever. A silent, permanent lying canary produced entirely inside a fully type-checked pipeline. Additionally, `data:` framing and the `\n\n` terminator are hand-written and unguarded, and there are **two** producers of `YardSnapshot` over this boundary (design:103 and design:104), which is a hand-maintained mirror by another name.

**Fix:** (i) delete the claim at design:167-168 and replace it with an explicit hop list that the per-car reviewers must trace; (ii) apply the schema module's runtime validator to the parsed wire payload in the view, so a divergence is a red test rather than a silent pass; (iii) replace the §6 shared-type test with a guard that can actually fail, e.g. a source scan asserting the `YardSnapshot`/`LaneState` declarations resolve to exactly one file, plus an ESLint `no-restricted-syntax` rule banning those declarations outside `src/schema.ts`; (iv) define the SSE event name as an exported constant in the shared module and assert both sides import it; (v) mandate that `/api/snapshot` and `/api/stream` serialize through one function, and test that both produce byte-identical output for a fixed snapshot.

### MAJOR-4. The SSE contract is underspecified in three ways, each of which produces a silent lie

**Location:** design:104-105, design:178, D7 at design:35.

**(a) "Emits only on change" is undefined, and both readings break.** design:105: "Emits only on change, so an idle yard is silent." Change of *what*?
- Reading 1, "the snapshot object differs from the last one sent": `YardSnapshot.asOf` (design:80) and `stale.ageMs` (design:75) change on every poll by construction, so nothing is ever silent and the stated benefit is fiction.
- Reading 2, "the adapter's data differs": then the `ok` to `stale` transition, which is caused by time passing and not by data changing, **never emits**. A client that connected while the lane was `ok` displays `ok` indefinitely while the server knows it is stale. That is `constitution.md:42-43` verbatim: "A stale board that looks live is the lying-canary disease this project's ancestors fought all year."

**(b) No heartbeat, and silence is deliberate.** The design makes an idle yard produce zero bytes and simultaneously requires the client to detect connection loss (design:178, D7). A half-open TCP connection (laptop sleep, wifi drop, an intermediary silently dropping state) leaves `EventSource` in `readyState === OPEN` indefinitely with no error event. With no periodic traffic, the client has no signal at all. §5's row "SSE disconnects, client shows 'disconnected, last good as-of X'" is therefore unbacked in the most common real disconnect mode. The design's design-review-mandated "observability reality" question (`CLAUDE.md:115-117`, "does the code actually expose what the design consumes?") answers NO here.

**(c) First-paint versus stream ordering race.** design:103 gives first paint via `GET /api/snapshot`; design:104 gives updates via SSE. The client does both. If the stream is subscribed first and an event arrives while the fetch is still in flight, the fetch resolves *afterward* with an **older** snapshot and overwrites the newer one. The board then renders stale state as current with no marking, until the next change event, which under reading (b) of (a) above may never come. No ordering rule, sequence number, or monotonic guard is specified. `CLAUDE.md:115` puts race conditions explicitly in this gate's scope.

**Fix:** (a) define change detection precisely, and make the `ok`/`stale` transition an explicit emit trigger regardless of data equality; (b) mandate an SSE comment heartbeat at a defined interval and a client-side watchdog that flips to "disconnected" after N missed heartbeats, and put the heartbeat interval on the wire so the client does not hardcode it; (c) mandate monotonic application by `asOf` (or add a monotonically increasing `seq` to `YardSnapshot`, which is more robust given the clock concerns in MAJOR-6) and a red test that a late-resolving first-paint fetch does not overwrite a newer streamed snapshot.

### MAJOR-5. Concurrency and state ownership: the state inventory is incomplete and §3.4's claim about it is false

**Location:** design:107-108, design:76, design:200-215, design:226-230.

design:107-108: "Holds the **only** long-lived mutable state in the system: `lastGoodSnapshot`, `lastPollAt`, `connectedClients`." That claim is false three times over.

**(a) There is no in-flight poll guard.** design:105 specifies "Poll loop on a fixed interval." If a poll outlasts the interval (a large file, a network drive, an antivirus scan on Windows, a disk stall), invocations stack. Two concurrent reads can complete out of order, and an **older** read then overwrites `lastGoodSnapshot`. `pollInFlight` (and the timer handle itself) are mutable server state the design does not enumerate, therefore they get no ledger row, therefore per `state-ledger.md:31-32` they are `LATENT_BUG` by default and nobody knows it. This is the "poll that overlaps a slow previous poll" lifecycle event, missing from design:205-212.

**(b) Per-lane `lastGood` has no owner.** design:76 requires `failed` lanes to carry `lastGood?: T; lastGoodAsOf?: string`. Producing that requires remembering the last successful result **per lane**. But design:98 describes the assembler as depending only on "adapters" and design:68-69 describes it as a pure combiner. Either the assembler is stateful (undisclosed mutable state, unledgered, owned by Car A) or the server must thread prior state into it (an interface between Car A and Car C that the design does not specify). Both readings are live; Car A and Car C will each assume the other does it. Note also that `lastGoodSnapshot` (whole-snapshot, server) and per-lane `lastGood` are two different granularities of the same idea, which is a second copy of something that can drift, and `constitution.md:48-49` forbids exactly that.

**(c) The ledger's ownership contradicts the living-contracts rule.** design:202-203 mandates the ledger be created "in the same commit as the code that creates the state," which restates `CLAUDE.md:243-245`. But design:229 assigns ledger instantiation to **Car C only**, while Car A creates the assembler's per-lane last-good state and Car D creates client-side state (rendered snapshot, connection status, reconnect counters, last-received `asOf`). If Car A lands before Car C, Car A's commit creates mutable state with no ledger to update, which its own reviewer must reject. If Car C lands first, it ledgers state that does not exist yet. Also: design:213 lists three fields, none of them client-side, yet design:209 enumerates "client reconnect" as a lifecycle event.

**Fix:** add `pollInFlight` and the timer handle to §7; specify a skip-if-in-flight (not queue) policy and a red test for it; name the owner of per-lane last-good explicitly and give the assembler a stated interface the way `Adapter&lt;T&gt;` has one; either make the ledger Car A's deliverable (first state landed) with later cars appending, or make it a §9 obligation on every car; decide explicitly whether browser state is in the ledger's scope and say so.

### MAJOR-6. `asOf` provenance is unspecified, which determines whether staleness ever fires at all

**Location:** design:45-46 (`AdapterResult.asOf`), design:74-75, design:95-96, design:139-141.

`asOf` appears on `AdapterResult`, on every `LaneState`, and on `YardSnapshot`, and the entire staleness mechanism (design:95-96, and Law 5's headline surface) is computed from it. The design never says where it comes from. Two readings:

- **Reading 1, the state file's mtime.** Then a `git checkout` or a clone rewrites mtime to *now*, so a state file whose contents are three days old reads as perfectly fresh. The demo fixture that D3 (design:31) and Law 7 (`constitution.md:54-55`) require is then permanently `ok`, which is a lie.
- **Reading 2, a timestamp field inside the file, written by the StateWriter.** Then a checked-in fixture reads as very stale, which is honest but makes the demo look broken; and a fixture hand-edited with a future timestamp (design:139-141 explicitly anticipates hand-editing) yields a negative age that never crosses the threshold, so the lane is permanently `ok`.

Both readings need mitigation and the design specifies neither. Related and unaddressed: `ageMs` is derived from wall-clock arithmetic, so a system clock step (NTP correction, DST on a naive local timestamp, a fixture from another machine) can make age negative or wildly large. `constitution.md:42-43` names the lying canary; this is the mechanism that produces it.

**Fix:** state the provenance explicitly, require in-file timestamps produced by the writer, require the adapter to reject or loudly mark future-dated `asOf` (negative age is not `ok`, it is `failed` with "state file timestamp is in the future"), and specify whether staleness uses wall clock or a monotonic delta from the last successful poll. Add a §6 test for a future-dated fixture and one for a clock step backwards.

### MAJOR-8. Two of the six named tests cannot be written as described

**Location:** design:192-196.

**(a) The atomicity test (design:193-194): "interleave a write with a poll; the poller never observes a partial file."** As specified this is a nondeterministic race test. Against a *correct* implementation it can never observe a partial file, and against an *incorrect* one it observes a partial file only probabilistically. `CLAUDE.md:96-97` requires "write the failing test, RUN it, confirm it fails for the stated reason." A probabilistically-red test cannot honor that rule, and the car assigned it will either flake, loop until it passes, or quietly weaken it. Worse, the test targets the wrong failure mode for this repo's platform: on Windows (and `board.ps1:41` establishes this repo is Windows-first) the realistic failure is not a partial read, it is `rename` over a destination another process holds open failing with `EPERM`/`EBUSY`. The design does not mention that at all, and it is the failure the atomic-write strategy actually risks here.

**Writable replacement, deterministic red:** (i) spy on the fs module and assert the destination path is never passed to any write/truncate call and that exactly one `rename(tmp, dest)` occurs. An in-place implementation calls `writeFile(dest)` and never `rename`, so this fails for exactly the stated reason before the fix and passes after. (ii) assert the temp filename cannot be mistaken for the state file by the adapter. (iii) assert `rename` over the destination succeeds while a read handle on the destination is open, run on Windows. (iv) keep any interleaving loop as a non-gating soak, never as the red.

**(b) The staleness test (design:192): "fake clock."** Faking the clock requires the assembler to accept an injected clock, but §3.3 gives `SnapshotAssembler` no interface block at all (contrast `Adapter&lt;T&gt;` at design:43-52, which does). design:95-96 says the assembler owns the threshold and the classification, without saying how either is supplied. The named test cannot be written until someone invents that constructor signature, and `CLAUDE.md:125-128` records three REJECT rounds paid for exactly this class of gap.

**(c) The shared-type test (design:195-196)** is covered in MAJOR-2(a): it cannot fail.

**Fix:** rewrite design:193-196 as above; give `SnapshotAssembler` a stated interface including its clock and threshold inputs.

## RULINGS ON THE FIVE OPEN QUESTIONS

**Q1, poll interval: config value or constant? Ruling: a constructor parameter with an exported default constant, plus a single `STARCAR_POLL_MS` environment override read once at startup. Not a config file.**
Reasons: (a) the staleness test (design:192) already requires injectable timing, so the parameter must exist for testability regardless of the Law 7 argument, which makes the constant-only option strictly more expensive, not simpler; (b) Law 7 wants a stranger to tune it and the marginal cost is one parameter; (c) §8 (design:217-221) excludes config infrastructure, so an env var is the right weight. **Additional binding requirement: the effective interval must appear in `YardSnapshot` and be rendered**, because Law 5 (`constitution.md:39-41`) makes the board's own operating parameters part of its self-knowledge, and because it is the number a stranger needs when the board looks slow.

**Q2, state file location and configurability? Ruling: default to a repo-root-relative path resolved from the process CWD, overridable by `--state &lt;path&gt;` and `STARCAR_STATE`, with the resolved ABSOLUTE path carried in `YardSnapshot` and rendered on the board. Separately: the live state file and the demo fixture must be two different files.**
Reasons: (a) Law 7 forbids hardcoded paths; (b) Law 5 requires the board to say what it is reading, and the single likeliest stranger failure mode is a board saying `failed: ENOENT` while the human cannot tell which path it looked at, which is a Law 5 self-knowledge failure with a one-field fix; (c) **the fixture separation is the substantive half of this ruling.** D3 (design:31) collapses "the live conductor state file" and "the in-repo synthetic demo fixture" into one artifact, and design:139-141 leans on it being git-tracked. As one file, every `starcar car set` dirties the working tree mid-train and every car's diff contains board noise, and the demo data is whatever state the last train left behind rather than a curated illustration. Split them: a curated `fixtures/demo-state.json` used by tests and by a documented demo mode, and a live state file whose tracked-or-ignored status the design states explicitly. Given the showcase thesis I would track the live file too, but that must be a stated decision, and the design must say whether the writer commits (it should not; a human does).

**Q3, second staleness threshold? Ruling: NO. One threshold in v0, and render `ageMs` numerically so the human performs the second-tier judgment themselves.**
Reasons: (a) the Healing Loop's anti-autoimmune edge (`the-healing-loop.md:73-78`): "imported ceremony that never caught anything here is a pruning candidate," and a second tier has caught nothing yet because nothing has run; (b) Law 3 (`constitution.md:30`): "Decoration that does not inform is cut," and two shades of yellow inform less than one number; (c) it costs nothing to defer, because `ageMs` is already on the wire (design:75), so a second tier is a pure view change later with zero schema churn. This is the one place in the whole design where I would spend *less* ceremony, and the author was right to ask.

**Q4, is `connectedClients` ledger-worthy? Ruling: YES. Ledger it.**
Reasons: (a) `docs/templates/state-ledger.md:11-14` names the exact failure mode of the opposite ruling: an audit of the ancestor project "found 99 latent instances of the same pattern, each individually 'too small to ledger'." The argument "it is only an implementation detail of the SSE layer" is verbatim the argument that produced ninety-nine bugs; (b) it is genuinely mutable, genuinely long-lived, and it changes across three of the four enumerated lifecycle events; (c) there is a real bug it catches: clients not removed on disconnect leak, and the leak is invisible until the "client reconnect" row forces someone to verdict it; (d) the cost of a ledger row is one line, and the cost of establishing the exemption precedent on the very first ledger this project writes is unbounded. **A ledger's first entry sets its threshold forever; set it low.**

**Q5, must "server says stale" and "I am disconnected" be visually distinct? Ruling: YES, distinct, and they must be capable of being displayed simultaneously.**
Reasons, and the decisive one is structural, not aesthetic: **the two conditions live at different scopes.** Staleness is a per-lane property produced by the assembler and carried inside `LaneState` (design:74-75); disconnection is a whole-board property of the client's transport and belongs to no lane. Merging them into "one degraded treatment" is a category error before it is a design choice: it forces the view either to paint a board-level condition onto four lanes or to hide a lane-level condition behind a board banner. Beyond that: (a) Law 6 (`constitution.md:49-50`) requires the board to SHOW disagreement rather than pick a winner, and D7 (design:35) already establishes these as two independent truths, so collapsing them in pixels discards a truth the data deliberately preserves; (b) both can be true at once (a lane went stale, *then* the connection dropped), and a single treatment cannot express that; (c) the actionability difference is total: "stale" means go look at the conductor or the adapter, "disconnected" means go look at the server or your network. Law 3 (`constitution.md:27-28`) is about shortening the path from state to decision, and these are two different decisions. **The design must also state where the disconnect indicator lives** (board chrome, not lane panels), which it currently does not.

## NEW FINDINGS

### MAJOR-1: the position / freshness composition is undefined, and §7's `never-polled` test cannot be written as described

`Lane` requires a `freshness` for every lane (design:135-140), and the Law 4 test requires every registered lane in every snapshot (design:391-392). A lane at position `dark` ("No equipment", design:100) has no adapter, so it never appears in the assembler's `results: Map&lt;...&gt;` (design:280) and no member of `Freshness` (design:115-119) means "polling does not apply here". Rev 1's union had exactly that member (`{ kind: 'no-adapter' }`, `git show d7b8463:docs/design/2026-07-21-v0-yard-skeleton-design.md`, line 80); rev 2 dropped it without comment while §6:383 still carries a "Lane has no adapter" row.

The defensible reading is that position now carries that fact and `dark` lanes sit at `never-polled` forever. If so, the view must suppress the freshness words in favour of the position words, and the design never says so. §7:394-396 says the opposite: "the view renders 'not yet read' ... never empty, never `dark`". "Not yet read" is false for a lane that will never be read (Law 1, `constitution.md:17`), and "never `dark`" contradicts `dark` being a valid position that some lane in the registry will hold on day one. A car cannot write that test against a registry containing a dark lane. This is the same class as rev 1's MAJOR-8, reintroduced by the fix.

Required: state the composition rule for position x freshness (which axis supplies the rendered words in each combination), and either restore a `not-applicable` freshness member or say in the contract that lanes without adapters are excluded from freshness rendering entirely. Then rewrite the §7 test so it is writable.

### MAJOR-2: `register` is a property of position only, so a failed lane in a `nominal` position renders calm

`register` lives on `PositionDef` (design:85) and nothing else in the contract produces one. §5.5:311 says "Register drives colour and weight" and §5.5:311-313 says "Nothing renders in an alarm register unless something is genuinely wrong". Take the v0 trains lane: position `live`, register `nominal` (design:98). The state file is deleted. §6:377 routes that to `failed` with code `file-missing`. Nothing in the document moves that lane out of `nominal`. The dispatcher's glance therefore reads calm over a lane with no data.

Note that §6 names a register explicitly on exactly one row, the no-adapter row (design:383, "`nominal` register"), and names none on the `failed`, `stale`, `schema-violation`, or `timestamp-in-future` rows. That asymmetry is the tell: the author is thinking in position-registers and the failure rows were never mapped.

This is Law 1 and Law 3 simultaneously, in the one scenario the project exists to render. It is the finding I would keep if I could keep only one.

Required: a stated derivation of the rendered register from position AND freshness, with the precedence rule written out, and §6 rows carrying their register.

### MAJOR-3: registry and vocabulary load failure is unhandled, and the detector turns one config fault into N false discoveries

§5.1:198 makes the lane registry and the position vocabulary "data files loaded at startup". §6's nine-row table (design:373-383) has no row for either file being missing, malformed, or unreadable. Three concrete outcomes, none specified:

- **Lane registry missing or malformed.** The assembler iterates the registry (design:164-165). An empty or failed registry yields zero lanes: a board rendering an empty yard. `constitution.md:36` names this failure verbatim: "a missing lane reading as 'no trains' is a lie of omission". Law 4 outranks everything below it.
- **Vocabulary missing or malformed.** Every position id resolves to no `PositionDef`, so every lane fires the detector (design:107-110) and the entire board paints `needs-attention` labelled "unrecognised position: '&lt;id&gt;'" N times. The design insists this path is "a discovery, not a bug" (design:382) and "never rendered as an error in the software" (design:110), so a single missing file is misattributed as N observations of new shop vocabulary. That is an instrument crying wolf, which `CLAUDE.md:246-252` calls worse than no instrument, and it hides the real cause on a Law 5 surface that is supposed to degrade loudly and correctly.
- **One bad register value in the vocabulary.** `PositionDef.register` is the closed union (design:85) but the vocabulary is a data file, so `register: "critical"` is expressible. The wire validator (design:228) would then reject the whole snapshot and §6:381 blanks the board on account of one bad row.

Required: §6 rows for registry-load and vocabulary-load failure with coded reasons; a rule that a vocabulary fault is reported as a vocabulary fault rather than fanned out through the detector; and a decision on whether a bad vocabulary row is fatal to the snapshot or is quarantined to its own position.

### MAJOR-4: the substitute Law 4 guard is not equivalent to the guarantee it replaces, and the design asserts that it is

§4.5:157-171 weakens rev 1's four required keys to a registry plus a test, states that Law 4 outranks Law 7, and then claims "Nothing can be silently dropped; the guarantee simply moved from the compiler to a test" (design:166-167).

The compiler guaranteed two things. (a) Every constructed snapshot contains every lane. (b) The lane set itself cannot shrink without a deliberate type edit that breaks every construction site. The replacement test, stated at design:391-392 as "every lane in the registry appears in every snapshot", covers (a) only. Delete a line from the registry file and: the build passes, the Law 4 test passes (the deleted lane is no longer registered, so the assertion is vacuously satisfied for it), and the lane disappears from the board with no user-visible trace. That is a lie of omission reintroduced by the fix.

The design applies exactly the right suspicion to the state file at §5.6:342-343 ("a human, a bad merge or a rebase can produce garbage the writer never saw") and none of it to the registry, which is the same kind of hand-edited file with a larger blast radius. Disclosed-but-wrong does not clear review, and §4.5 is a disclosure attached to an overstated guarantee.

Required: a completeness guard on the registry itself. Cheap options that do not re-hardcode a board schema: pin the expected lane-id set for this deployment in a fixture-backed test, ledger the lane count with old to new arithmetic, or have the server render "registry declares N lanes" in chrome so a shrink is visible on the surface it damages.

### MAJOR-5: the domain payload is undefined, and §12's Minor-10 disposition claims a vocabulary that exists nowhere in the contract

§4 specifies the envelope exhaustively and never specifies what a lane actually carries. `Lane.freshness` is `Freshness&lt;unknown&gt;` (design:139). `Yard` appears only in prose at design:205 ("carries `trains` and `gates` in v0"). There is no `Train`, no `Car`, no `Gate` type anywhere in the document. Rev 1 was more specific here, not less: it named `LaneState&lt;Train[]&gt;`, `LaneState&lt;Gate[]&gt;`, `LaneState&lt;Ticket[]&gt;`, `LaneState&lt;Fuel&gt;` (`git show d7b8463:...`, lines 82-85).

Consequences that block a plan-writer:

- Car 2 must invent the schema for the actual product, car 5 must render it, and car 3's CLI must write it, with no shared definition in the reviewed design. Cars 2, 3 and 5 each see only their own task per the brief template (`docs/templates/car-brief.md:18-19`), so their interface blocks cannot be made to agree from this document.
- design:328 shows `starcar car set c2 rolling --sha abc1234` and `starcar gate set plan-review reject`. `rolling` and `reject` are values in vocabularies the contract never defines.
- §12:540 disposes of rev 1's Minor-10 as "Now vocabulary, not enum - same open-set treatment as positions." No such vocabulary exists. `PositionDef` (design:82-87) is a lane concept carrying `surfacesData`; it cannot serve car states. The closure table asserts a fix the body does not contain.
- §11:506-510 makes the StateWriter the v0 embodiment of Law 2, using "marking a train held" as its worked example. There is no `held` value in any defined vocabulary, so the mechanism the design offers for the second-ranked law cannot currently express the law's own example.

Required: define `Yard`, its trains/cars/gates, and the car-state and gate-verdict vocabularies with the same closed/open ruling §2 applies to positions, or explicitly declare the payload out of scope for v0 and accept that the board renders nothing but lane chrome (which would contradict §1 and Law 3).

### MAJOR-6: two sources for one fact, position, with no precedence rule (Law 6)

design:199-200: "The lane registry declares which lanes exist and each one's current position; it changes on deploy, not at runtime."
design:107: "a position id present in **the state file** or lane registry that resolves to no `PositionDef` ..."

If a position id can arrive in the state file, then the conductor can change a lane's position at runtime and design:199-200 is false. If it cannot, design:107 is false. One of the two sentences is wrong, and the design does not say which. `constitution.md:49-50` rules on precisely this shape: "Where two sources disagree, the board SHOWS the disagreement rather than picking a winner silently." The design invokes Law 6 four times and does not apply it to the one seam it created.

This is the same class as rev 1's MAJOR-3 (an internal contradiction between two sections), which §12:528 records as dissolved.

Required: name the owner of a lane's position, say what the other source does with a position field if it carries one, and if both may set it, specify the disagreement rendering Law 6 demands.

### MAJOR-7: the rendered age has no update mechanism, and both available readings break a law

§6:379 and §7's ruling record (design:544) require `ageMs` rendered numerically. §5.3:254-257 excludes `ageMs` and `asOf` from change detection and includes `freshness.kind`. So after a lane transitions `fresh` to `stale`, nothing in the snapshot changes again: same data, same kind, and the two fields that do change are excluded. Zero further emits.

- Reading A: the browser renders the last emitted `ageMs`. The board displays "stale, 15s" for the next forty minutes. A specific wrong number on a truth surface is Law 1, and `constitution.md:42-43` is the exact text ("A stale board that looks live is the lying-canary disease...").
- Reading B: the browser computes age locally from `asOf` and its own clock. That is the UI computing its own state, which `constitution.md:47-48` forbids by name, and it reintroduces browser-to-server clock skew that §5.7:359-361 works hard to control on the server side.

The design chooses neither. Note that this is not the heartbeat's job: the heartbeat detects transport death, not data ageing, and D7 (design:62) is right that they are separate truths.

Required: pick a mechanism and write it down. The cheapest honest one is to emit on `ageMs` bucket changes or on a fixed tick while any lane is stale, so the number on screen is always a number the server issued.

### MAJOR-8: `demoMode` suppresses a truth surface (Law 1), and contradicts the gating-matrix template in the train that instantiates it

design:367-369: "Demo mode renders a persistent DEMO banner and treats age as informational, so a checked-in fixture does not read as a broken board." `demoMode` rides the wire at design:150.

`docs/templates/gating-matrix.md:23` is the template's single worked example row:

&gt; `| Example: staleness banner | data age &gt; threshold | never (truth surface) | fresh fetch | DELIBERATE, no override | Banner_Stale_... |`

§8:447-449 lands `docs/contracts/gating-matrix.md` in this train with the staleness banner as one of its four rows. As designed, that row's "Suppressed when" column reads "demo mode", against a template whose only example says "never (truth surface)". Law 1 grants no convenience exemption, `CLAUDE.md:52-55` names the pattern ("a lie with good intentions", "Law 1 does not grant exemptions for optimism"), and this repo publishes its screenshots.

The alarm is also unnecessary. The fixture reads stale because its `writtenAt` is old; stamp it. See my ruling on Q4 below.

Required: remove the suppression. Keep the DEMO banner, which is an honest additional disclosure rather than an override.

## RULINGS on the five open questions (§13)

**Q1. Registers closed at three, or does `in-progress` collapse into `nominal`? KEEP THREE.** Collapsing forces `under-construction` to render identically to `live`, which fails Law 3's glance test in the opposite direction from over-alarming: the dispatcher can no longer tell "someone is on it" from "it is done and quiet", and those imply different actions. Three is also the smallest set that lets MAJOR-2 be fixed without inventing a fourth. Caveat: this is the wrong argument to be having until the freshness-to-register derivation exists, because today the register count is not what determines what the board paints.

**Q2. `positions` on every snapshot, or fetched once? EVERY SNAPSHOT.** A separately fetched vocabulary is a second copy with an independent lifetime, so a client that fetched it at 09:00 renders 09:40's lane positions against 09:00's vocabulary and the drift is silent. That is the mirror class this repo has a scar for (`CLAUDE.md:163-174`), and it would also break the byte-identical parity test at design:411-412 by splitting the payload. The cost is four rows of JSON on loopback at 1 Hz, which is nothing, and MAJOR-7's fix will reduce emit frequency rather than raise it. If size ever matters, send a vocabulary hash and let the client refetch on change; do not solve a problem you do not have.

**Q3. Is browser-side state ledger-worthy, or ledger over-reach? IN SCOPE, BOUNDED.** Ledger exactly the three fields named at design:443-444 (rendered snapshot, connection status, last applied `seq`) against the lifecycle events that actually touch the browser (client reconnect, server restart). The design's own argument is correct: listing "client reconnect" as a lifecycle event and then verdicting it against no client-side rows is theatre, and `lastAppliedSeq` surviving a reconnect is precisely the state-survives-a-lifecycle-event bug pattern the ledger exists for (`CLAUDE.md:259-263`). Bound it explicitly: transient render caches and DOM node handles are NOT ledgered, so the first entry sets a threshold at "state whose survival across a lifecycle event changes what the user sees", not at "every variable in the view".

**Q4. `demoMode` staleness suppression: legitimate accommodation or Law 1 breach? BREACH. Remove it.** This is MAJOR-8. The fixture reads stale because it is stale; suppressing the alarm makes the board say something false in the one configuration most likely to be screenshotted and published by a project whose thesis is that its surfaces can be trusted. The correct fix is the one the question itself proposes: stamp `writtenAt` at fixture-load time so the demo data is genuinely fresh, which also exercises the real freshness path in the demo instead of bypassing it. Keep the persistent DEMO banner (it is an additional honest disclosure, and Law 5 likes it). If someone later wants a deliberately-stale demo to show the staleness banner working, that is a second fixture, not an override. `docs/templates/gating-matrix.md:23` already ruled on this before the question was asked.

**Q5. Car 1 (toolchain + CI) first, or after car 2? FIRST, WITH A CONDITION.** The attack I was asked to run here fails: CI on an empty repo is not vacuous, because what car 1 asserts is not "the product works" but "the pipeline that ships it exists and runs", which is the precondition `CLAUDE.md:241` makes every later car's verification claim depend on. Ordering it second means car 2's report says "verified" while the only evidence is a local run. The condition: car 1 is not done when CI is green, it is done when someone has WATCHED it go red. Fault-inject (break a test, push, observe red, revert) and put the run URL in the report, per the GUARD CHECK duty at `docs/templates/car-brief.md:65-67`. A green light wired to nothing is worse than no light. Second condition: move the ESLint single-definition rule to car 2 or accept that car 1 ships it green-on-arrival (Minor-5).

{'agentId': 'a56d4b46b4589a001', 'description': 'Adversarial design review v0 round 2',
 'isAsync': True, 'status': 'async_launched', 'resolvedModel': 'claude-opus-4-8[1m]',
 'outputFile': '...\\tasks\\a56d4b46b4589a001.output', 'canReadOutputFile': True}
```

`status: async_launched`. That is what the Task tool returns, and therefore what `PostToolUse:Task` receives, and it is received **at launch**, not at completion. There is no body, no outcome, no envelope. The actual completion arrives much later as a synthetic `user` message carrying `&lt;\task-notification&gt;&lt;task-id&gt;...&lt;status&gt;completed&lt;/status&gt;&lt;result&gt;...`, which fires no `PostToolUse` hook at all.

This is not speculation about the runner. It is the reason `Land-Verdict.ps1` is written the way it is: `Get-ResultBlockForTask` (`scripts/Land-Verdict.ps1:78-116`) scrapes `&lt;task-id&gt;` and `&lt;result&gt;` out of the transcript precisely because there is no tool result to read. The existing implementation is evidence against the design's replacement mechanism, and the design does not notice.

Consequences as written: the harness would emit a `returned` event at launch for every dispatch, with `status: async_launched` and no body. Under §8's fold, no car would ever appear to be rolling, every dispatch would be instantly terminal, and every artifact body would be empty. That is a confident falsehood on a status surface generated by the design itself (`constitution.md:13-17`).

The correct hook event for "a subagent finished" is `SubagentStop`, not `PostToolUse:Task`. Note that `SubagentStop` gives you a transcript path, not a body, so the extraction machinery does not go away; it changes trigger. The design must name the real lifecycle and state where the body comes from, because §4.4's whole artifact depends on it.

**This one finding invalidates §7's central claim.** Everything §7 says about hooks being mechanism rather than vigilance is correct in principle and wrong in mechanism.

### MAJOR-2. Nothing in the design detects a MISSING artifact. The class has moved one hop, not closed.

You asked me to attack this directly, so: the stated class (design:18-20) is *"artifacts survive only if the conductor remembers."* §6 verifies hashes of artifacts that exist. §5's index is generated from artifacts that exist. §4.3's "loud failure" fires only on an envelope that arrived and could not be parsed. **An absence is invisible to every mechanism in this document.** A hash cannot report on a file that was never written.

The only reconciliation is `Land-Artifacts -Sweep` (design:184-187), and the design says *"The `/goodnight` ritual runs it."* Two problems:

1. `/goodnight` is a human-invoked skill. `.claude/skills/goodnight/SKILL.md:3` reads *"Invoke when the owner signs off for the day or asks to close the session cleanly."* That is a **procedure**, which the Healing Loop ranks above vigilance but below a test and below a structural impossibility (`docs/the-healing-loop.md:36-40`). The class does not close; it moves from vigilance to procedure. That is a real improvement and it should be claimed as one, but the design claims closure (design:179).
2. Worse: `.claude/skills/goodnight/SKILL.md` at this base contains **no sweep step**. Steps 1 through 7 are triage, push sweep, state checkpoint, board pass, CI disposition, yard-status close, worktree prune. The design's backstop is not written anywhere, and §10's car table assigns nobody to write it (see Minor-2).

And the failure modes you listed are all real and all unaddressed: a hook that is not configured (a stranger's clone has no `.claude/settings.json` producer entry), a hook whose command fails (note that every existing hook in `.claude/settings.json` uses the pattern `if ! command -v entire &gt;/dev/null 2&gt;&amp;1; then exit 0; fi`, a **silent no-op** when the tool is missing, which a StarCar producer copying house style would inherit), a runner that fires no hooks at all, or `sh` not being on PATH on this Windows box.

The honest closure requires a positive reconciliation: the `dispatched` event is the ledger entry, and something must periodically assert "every `dispatched` has a terminal event or an explicit unknown." The design has the `dispatched` event that makes this possible and never uses it that way.

### MAJOR-3. Ghost dispatches. No reaping, no timeout, no liveness anywhere. (design:64-65, 205-206)

§4.1 argues in-flight state should be derivable: *"a dispatch with a `dispatched` event and no terminal event **is** in flight, with no separate status field to go stale."* §8 makes that the board's data: `A car is rolling | dispatched with no terminal event` (design:205).

Kill the session, sleep the machine, SIGKILL the process, or let the agent die in any way that produces no terminal event, and the store contains a `dispatched` with no terminal event **forever**. The board shows a car rolling that died three days ago. There is no timeout, no heartbeat, no reaper, no max-age, and no "unknown" outcome in the design.

The design has traded a status field that can go stale for a *derivation* that goes stale, and derived staleness is worse because there is no field to look at and correct. `constitution.md:13-17`: *"a train as idle when it is burning tokens is worse than no board at all."* The inverse (a train burning when it is dead) is the same defect, and Law 1 requires *"Unknown states render AS unknown, honestly"* (`constitution.md:17`).

The parked yard design's freshness machinery does not save you here: `Freshness` describes the **adapter's** last successful poll of the store (`docs/design/2026-07-21-v0-yard-skeleton-design.md:171-179`). A store that is being appended to for other dispatches reads perfectly `fresh` while carrying a three-day-old ghost. Per-dispatch liveness is a different axis and nothing in either document owns it.

This is directly relevant to open question 1: **append-only plus no reaping is strictly worse for in-flight truth than a mutable record**, because a mutable record at least has somewhere to write "timed out."

### MAJOR-4. The hash does not cover the fields the board renders. Fault-injected: the verifier says OK. (design:38, 116-140, 152-153)

I ran the existing verifier on both landed verdicts: `2 verdict file(s) verified: every body matches its claimed hash`, exit 0. Then two injections, in a scratch copy:

**Injection A (body).** Changed `# VERDICT: REJECT` to `# VERDICT: APPROVE` in the body of `docs/reviews/2026-07-22-design-v0-round2-REJECT.md`. Result: `MISMATCH`, `claimed: 32473a92...`, `actual: 874c2a63...`, exit 1. **The guard fires.** Good.

**Injection B (header).** Changed only the header line `**Verdict: REJECT - 8 Major, 15 Minor, 5 Notes**` to `**Verdict: APPROVE - 0 Major, 0 Minor, 0 Notes**`, leaving the body untouched. Result:

```
OK           ...\faultinject\headertamper.md

1 verdict file(s) verified: every body matches its claimed hash.
EXIT=0
```

A file whose header declares APPROVE with zero Majors over a body that argues REJECT with eight passes its own verifier cleanly.

The design inherits this defect and makes it worse. §4.4's front matter carries `outcome`, `findings`, `gate`, `target`, `base`, `model`, `cost` and `at`, and `body_sha256` is described (design:162) as *"over its verbatim body."* §5:152-153 then says the index, which is what a reader and the board consume, is **generated from front-matter**. So the exact fields that reach the board are the ones outside the hash's coverage.

§3's first table row therefore states a defence the mechanism does not provide: *"The conductor alters a verdict about its own work | Body SHA-256 stamped at extraction, recomputable by anyone | -"* with an empty "Not defended" cell. That empty cell is false. A conductor who wants a REJECT to read as an APPROVE does not need to touch the body at all.

There is a related ambiguity that must be resolved before a spec is written: §4.3 has the agent emit the envelope inside its report, and §4.3's parsing rule ("matched as a LAST-occurrence pair") implies the producer cuts it out. If the envelope is stripped before hashing, the front matter has no verifiable link to anything. If it is retained inside the hashed body, the front matter is a derived mirror that a verifier could re-derive and check, but the design never says the verifier does that. The document is readable both ways, which is a finding in its own right (`CLAUDE.md`, spec review: any requirement readable two ways).

Law 6 (`constitution.md:47-49`) is the law being broken: the front matter is *"a second copy of anything that can drift"* from the body, with no check that they agree.

### MAJOR-5. The fold has no identity, no ordering, no deduplication, and no supersession. (design:56-65, 151, 196-214)

You asked whether the fold can actually reconstruct state. As specified, no. Four concrete holes:

**(a) Identity.** §5:151 gives paths as `docs/artifacts/YYYY-MM-DD/HHMMSS-&lt;kind&gt;-&lt;slug&gt;.md`. A timestamp is not an identity: two events in the same second collide, and `&lt;slug&gt;` is never defined. §4.4's `dispatch: &lt;opaque runner id&gt;` is the only candidate identity and the design explicitly declines to constrain it. `agentId` values in this runner (`a56d4b46b4589a001`, `a9fa2727d341bde1b`, `a1bccdf2b25d3bd94`) look session-scoped; the design states no uniqueness guarantee across sessions, machines, or clones, and the fold's central operation ("`dispatched` with no terminal event") is a join on that key.

**(b) Duplicates.** §7 explicitly foresees the sweep landing events the hook may already have landed (*"lands anything missing"*, design:185), and the transcript's own `&lt;\task-notification&gt;` note says *"the same task-id may notify more than once"* (which `Land-Verdict.ps1:113-115` already handles by taking the last). §4.1 says nothing is ever mutated. So the store can legitimately hold two `returned` events for one dispatch, from two producers, with different `at` timestamps, possibly different bodies. The design specifies no dedup and no last-wins rule, so the fold's answer to "did this car finish" depends on iteration order.

**(c) Supersession, and this one implicates Law 2.** §8 makes `A train is held` a conductor `intent` event (design:213). Under append-only with no supersession rule, a hold can never be **released**: the `intent: held` event is still in the store forever. §4.2's `ruling` row mentions *"what it supersedes"* for rulings, and no equivalent for `intent`. `constitution.md:21-23` requires the board to *"never resist an override"*, and a mechanism where the dispatcher can set a hold but not clear it resists one. The very law the intent channel exists to serve is the one it cannot express.

**(d) Ordering and clock source.** `at` is stamped by whom? A hook-landed event gets wall-clock at hook time; a sweep-landed event gets a timestamp reconstructed from a transcript. Same field, two producers, two clocks, and the fold's "no terminal event" test is order-sensitive. The design's own §4.4 example (`at: 2026-07-22T00:26:56Z`) does not say which.

§4.1's three arguments for append-only are all good arguments. None of them survive contact with a fold whose semantics are undefined. Answer these four before a car writes a reducer.

### MAJOR-6. The fuel gauge's data source does not exist as described. (design:142-145)

§4.4 states: *"`cost` is not decoration: it is the **fuel gauge's data** ... The runner reports tokens, tool calls and duration per dispatch; capturing them here makes the fuel lane real rather than hypothetical, from artifacts we already produce."*

I went looking for that report. It is not there.

- The Task tool result contains no cost fields at all (see MAJOR-1's dump: `agentId`, `description`, `isAsync`, `status`, `resolvedModel`, `prompt`, `outputFile`, `canReadOutputFile`).
- The per-subagent metadata file `...\subagents\agent-a56d4b46b4589a001.meta.json` contains exactly: `{"agentType":"car","description":"...","toolUseId":"...","spawnDepth":1,"model":"opus"}`. No cost.
- The `&lt;\task-notification&gt;` block carries `task-id`, `tool-use-id`, `output-file`, `status`, `summary`, `note`, `result`. No cost.
- A grep of the whole project transcript for `totalTokens`, `totalToolUseCount`, `totalDurationMs`, `wasInterrupted` returns **zero** matches for each.

What does exist is per-assistant-turn `usage` objects inside the subagent transcript, with **five distinct counters**, for example the final turn of the round-2 reviewer:

```
{'input_tokens': 2, 'cache_creation_input_tokens': 1830, 'cache_read_input_tokens': 97453,
 'output_tokens': 15957, 'server_tool_use': {...}, 'service_tier': 'standard', ...}
```

So `tokens: 99390` in §4.4's example is not a reported number, it is an undefined sum of five counters that are **billed at different rates**. A fuel gauge that the operation is budgeted against (`README.md:44`, issue #1's fourth surface) cannot answer "how much of my window did this burn" from one scalar that conflates cache reads with output tokens. `tool_uses` and `duration_ms` are likewise derivations (countable and subtractable from the subagent transcript, but nobody reports them).

This matters at design rung because the design uses the claim to justify a whole surface going from hypothetical to real. As written, a spec writer specs a field, a car goes looking for the number, finds five, and honest-stops. Catching it here is one dispatch; catching it at the car's compile wall is the failure mode `CLAUDE.md`'s plan-review scar describes. Either define the derivation and its semantics (and say plainly that it is a derivation, which Law 6 permits an adapter to do), or mark `cost` optional and out of scope for this train.

### MAJOR-7. Automated, unredacted, verbatim publication of every dispatch body to a public repo is a scope expansion the owner has not ruled on, and it contradicts an in-repo ruling.

`CLAUDE.md`'s HARD INVARIANT and `docs/setup.md:40-48` record the owner's "full monty" decision: **session transcripts** publish to the Entire checkpoint branch. That is a decision about the transcript mirror.

This design is a different surface. It commits `&lt;agent's report, byte-for-byte&gt;` (design:139) into the **main repo tree**, on **every dispatch**, driven by **hooks with no human in the loop** (design:174-177), with no redaction step and no review before publication. The existing scripts required a deliberate conductor action per artifact with an explicit output path (`Land-Verdict.ps1:39-51`, seven mandatory parameters). This design removes that action entirely and the design does not name the removal as a decision.

It is not hypothetical. The two artifacts already landed by the manual path publish the operator's absolute home directory three times:

- `docs/reviews/2026-07-22-design-v0-round2-REJECT.md:25` and `:27`
- `docs/reviews/2026-07-21-design-v0-round1-REJECT.md:296`

each containing `C:\Users\Chris\AppData\Local\Temp\claude\C--Users-Chris-git-starcar\...`. Every brief in this shop mandates base-verification in a named worktree, so **every future car report will contain that path by construction**, and the sweep will publish all of them.

And this repo has already ruled against exactly this disclosure class, in the parked yard design at `docs/design/2026-07-21-v0-yard-skeleton-design.md:582-584`:

&gt; `statePathDisplay` is CWD-relative or home-collapsed rather than a raw absolute path, because this repo publishes screenshots and an absolute path discloses the operator's directory layout for no Law 5 benefit a relative one does not also deliver.

The board is forbidden from rendering the absolute path; the artifact store publishes it verbatim on every dispatch. That is a live inconsistency between two design documents in the same repo.

Beyond paths: a car report quotes its brief, and briefs quote plan text, cross-car warnings, and sometimes verbatim owner instructions. Verbatim-by-construction is the right principle for **fidelity between the agent and the record**; it is not automatically the right principle for **fidelity between the record and the public**. §9's out-of-scope list does not mention redaction and §11 does not ask about it. This needs either an explicit owner ruling recorded in the design, or a redaction/quarantine step (land verbatim, publish reviewed), or at minimum a home-collapse normaliser matching the ruling already made.

I am raising this as Major rather than Minor because it is undisclosed, it is irreversible once pushed to a public repo, and the design's own governing document opens with a HARD INVARIANT about what becomes world-readable here.

## RULINGS ON THE SIX OPEN QUESTIONS

**Q1. Append-only events vs a mutable per-dispatch record. KEEP append-only.** The three arguments at design:60-65 are sound and history genuinely is the product here. But the honesty is not free the way §4.1 implies: as specified, this is not "state becomes derivable," it is "state becomes underivable," because MAJOR-5 leaves the fold without identity, ordering, dedup or supersession, and MAJOR-3 leaves in-flight state permanently wrong. Note the sharpest form of the trade: **append-only plus no reaping is strictly worse for in-flight truth than a mutable record**, because a mutable record has a field where "timed out" can be written. Keep events; owe the spec a section titled "the fold," answering those five questions before any car writes a reducer.

**Q2. Agent envelope vs producer-derived metadata. BOTH, split by who can actually know.** The design frames it as a choice and it is not. The producer can observe `base` (git), `model` (`resolvedModel` in the tool result, `model` in the subagent meta file), `at`, `dispatch`, `gate` and `target` (from the brief it just launched), and any cost derivation. Asking the agent for those is asking a stranger for facts you already hold, and every one of them becomes an agent-drift surface for no gain. Only `outcome` and `findings` require the agent's judgement of its own work. So: producer-derived for everything observable, envelope for the two judgement fields, and those two hashed together with the body (MAJOR-4) so the assertion is pinned to the text that justifies it.

On "does the cross-check catch drift": **no, and as written it is a lying-instrument risk.** design:112-114 says *"Counts are cross-checked against the body where the body is structured enough to count."* There is no defined finding format in a verdict body. Counting `Major` in this document would find the word in my headings, in "any Major = REJECT" quoted from the calibration, in cross-references, and in the phrase "seven Major findings." The escape hatch ("where the body is structured enough") makes the check unfalsifiable: it can always claim the body was not structured enough. `docs/the-healing-loop.md:64-65` rules on this directly: *"an instrument that cries wolf is worse than no instrument."* Either define a machine-countable finding format that briefs mandate (a heading grammar, or the envelope carrying finding ids), or delete the cross-check and say plainly that finding counts are the agent's assertion, hashed with the body that supports it, and rendered as such.

Also rule the failure path honestly: design:110-111 says a missing envelope is recorded as `error` with the raw body preserved. That is the right shape and it is Law 4 correct, provided the artifact still lands with its body and the board renders "no envelope" loudly rather than the artifact silently not existing. Make that explicit, because MAJOR-2 means the alternative (nothing lands) is currently indistinguishable from nothing happening.

**Q3. Is `verdict` a facet of `returned`, or two events? ONE artifact. Keep it.** The worry in the question (a garbage reviewer produces no verdict at all rather than an explicit failure) is answered by the outcome vocabulary, not by splitting. Splitting creates two records of one event that can disagree, which the design correctly cites Law 6 against (design:77-79). The fix is to state that **`kind` is set at `dispatched` from the brief (dispatch intent) and `outcome` is set at completion (result)**. A reviewer dispatch that returns garbage then folds as `kind: verdict, outcome: error`: a verdict-shaped hole that the board can render as "reviewer failed to produce a verdict," which is exactly what the question asks for, with one record.

**Q4. Does retiring `docs/reviews/` break `README.md:20-21`? NO, if the move and the index land in the same commit.** The promise is *"the review verdicts and REJECT records committed in-repo as they happen"* (`README.md:19-21`, verified). Relocation with git history preserved keeps that promise; the north star already forbids the window you are worried about, since the commit that invalidates a document updates it in the same commit. The window is a car-brief constraint, not a design problem. **But the question points at the wrong line.** The README line this design actually falsifies is `README.md:46-47`, which lists *"a conductor-maintained state file"* among the adapters (Minor-4), and §10 assigns no README owner at all.

**Q5. Is the body hash security theatre? Argued, then ruled: KEEP, WITH HONEST REFRAMING AND WIDER COVERAGE.**

The case that it is theatre, made properly. The conductor controls the producer that computes the hash, the hash function, the verifier that checks it, the CI configuration that runs the verifier, and the git history that stores all four. A conductor who wants a different verdict does not edit a file and get caught; they re-run the producer against an edited transcript, or change one line in `Get-Sha256`, or delete the CI job, or force-push. §3 already concedes the hash says nothing about whether the agent was honest and nothing about a forged transcript. Against the only threat it claims, the party it defends against is the party that owns every component of the defence. And empirically it is worse than that: MAJOR-4 shows it does not even cover the fields the board reads, so today the defence fails against a casual edit, never mind a determined one. In its current form it is not theatre, it is a **lying instrument**, which this project's own doctrine ranks below having nothing.

Now the ruling. Do not cut it. Reframe row 1 of §3 from *"The conductor alters a verdict about its own work"* to what a hash can actually detect: **the artifact changed after extraction, for any reason, including innocent ones**. That threat is not hypothetical and this repo has hit it three times already: an ANSI-decoding regression, a BOM/CRLF write mismatch, and a separator collision, all caught or catchable by exactly this mechanism (issue #7's defect list; `Land-Verdict.ps1:69-73, 133-137, 170-176`). A hash that catches encoding drift, merge mangling, and edit-in-passing on a public record is worth its cost. Then make the reframed claim true by extending coverage over the envelope as well as the body.

And say out loud what the actual defence against a dishonest conductor is, because the design half-knows it: **publication**. §3:44-46 files "the verdicts are public and the next reviewer reads them" under the wrong row. Entire's checkpoint branch is an independently-written second copy of the same body that the conductor does not solely control, which is the only thing here that constrains a determined conductor at all. That belongs in row 1 as the real defence, with the hash demoted to what it is: an integrity check against accident and drift.

**Q6. The bootstrap paradox. Land this train's own verdicts with the CURRENT scripts, into `docs/reviews/`, and let car 3 migrate them with the other two.** Holding a verdict until the store exists is exactly the vigilance class the train is built to close, applied to the single most load-bearing review record in the repo (issue #7: *"The harness that enforces the process was exempt from the process. ## WHAT IS GOOD, AND WHERE MY ASSIGNED ATTACKS FAILED

**§3 is a genuinely good section and I want it kept.** A trust model with two ✗ rows written by the author, before review, is the honest-failure framing the Healing Loop calls its load-bearing wall. My MAJOR-4 says the table is wrong; it is wrong in a direction the table's own honesty made easy to find.

**§8's intent/facts split is the best idea in the document.** "The conductor declares INTENT; the process emits FACTS" removes the shared field rather than adding a precedence rule, which is a structurally better answer than the one the yard design spent two rounds converging on. It survives every attack I made on it, and I would not want it reverted.

**Attack H (Law 7 / vendor independence) mostly FAILED, and I am saying so plainly.** I went looking for Claude Code's model leaking into the schema and mostly did not find it. `kind` / `outcome` / `gate` / `target` / `base` / `body` / `body_sha256` are runner-agnostic. The `dispatched` and `returned` lifecycle is a property of "a shop that dispatches workers," not of this vendor. §7:191-194 confines the vendor transcript format to the producer and documents it as a known coupling with a stated blast radius, which is precisely what Law 7 asks. The `PreToolUse`/`PostToolUse` lifecycle is named as the **producer's** trigger, not as part of the schema. My only residual is Minor-9 (the cost triple, optionality), and the identity half of the concern is folded into MAJOR-5 where it belongs. The claim "the schema is the product, producers are adapters" is substantially true as written.

**Attack J (claim truth about the existing scripts) FAILED.** All three claimed defects are fixed, in code, at the lines the design implies, and the fixes work: verifier exit 0 on both landed files, exit 1 with a genuinely different hash on a tampered body. The design's account of its own history is accurate and its scars are attached correctly. What the account omits is the header gap, which is MAJOR-4 and is a different defect from the three it claims.

**Attack G (right-sizing), both directions, ruled.** Against: this shop is about to spend 14 dispatches on tooling to observe a process that has not shipped one line of product code, while the product train sits parked at rev 3 having already burned two REJECT rounds, and `docs/the-healing-loop.md:73-78` warns young projects to watch the autoimmune edge hardest. For: issue #7 is an owner ruling that the harness **is** product, `CLAUDE.md`'s right-sizing rule reserves the full ladder for *"new subsystems, rewrites, cross-boundary contracts"* and this is all three, and the harness is the data source every later gate and the board itself inherit, so a defect here is the most expensive kind this repo can have. **Ruling: the full ladder is correct, the car count is one too many.** Three cars (schema + validator + index + verification; producer + hooks + sweep + brief and agent updates; migration + CI), three reviewers, roughly 11 dispatches. Minor-10, not a Major. One real risk worth stating: designing a schema before its only consumer exists means the consumer is imagined, so rev 4 of the yard design should be treated as the schema's first real review, not as a downstream inheritor.

**Attack C, first-hand compliance evidence, as requested.** Emitting the envelope below was awkward in four specific ways, all of which are the design's problem and not mine:

1. **I had to count my own findings and nothing stops me writing zeros.** §4.3's stated purpose is removing the conductor's self-asserted summary of a review of the conductor's own work. It relocates that assertion to the agent. The agent is not disinterested about its own output either, it is just differently interested. This is a real improvement (the reviewer is at least the party who did the work) and it is not the removal of an assertion, which is how §4.3 reads.
2. **The parser and I had to negotiate.** I quote the opening sentinel in Note-5 and in MAJOR-4's discussion. I chose not to write a matching close-sentinel anywhere except the real envelope, specifically so the LAST-occurrence rule would pick mine. I reasoned about the producer's parser in order to comply. A reviewer whose report ends with a worked example of a malformed envelope would have that example captured as the artifact, and would have no reason to suspect it. The collision the design calls "implausible in prose" occurred in the first document that ever complied.
3. **The schema is undefined at the edges.** Is `round` mandatory? The example has it. Must strings be quoted? `target` is a path with slashes, `outcome` is a bare word, `findings` is inline flow: three YAML conventions in six lines, and nothing says whether the parser is a YAML parser. I guessed, consistently with the example, which means the example is the spec.
4. **My report is an async task result.** The envelope will live inside a `&lt;result&gt;` block that no `PostToolUse:Task` hook will see (MAJOR-1), reaching the store only via the sweep, which nobody has written into `/goodnight` yet (MAJOR-2). The mechanism I am dogfooding would not, today, capture this artifact at all.

None of that makes the envelope the wrong idea. Metadata belongs with the party who knows it. It makes the envelope a mechanism that needs the producer to own everything it can observe, a defined finding grammar or an honest admission that counts are an assertion, and the hash extended to cover it.
